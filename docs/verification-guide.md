# Verification guide: golden-notebook practice

Per SPEC §6. A workflow skill that encodes a
computation is not done until its marimo notebook in `verification/`
runs green headless (`python verification/x.py`, nonzero exit on
assertion failure; PEP 723 inline deps so `uv run` works in CI).

## Practice distilled from this build

- **Assert against recipe concepts, not constants you invent**: the
  notebook mirrors the recipe's expected values and states so; when
  recomputation drifts from a recorded anchor, the golden fails and
  someone learns something (that is the point).
- **Fixtures**: synthetic ones are generated by a committed script
  with a fixed seed and never committed as artifacts; real-data
  fixtures are scripted, cached subsets (see
  ocean-science/verification/fixtures/fetch_ecco_2010.py), never
  committed above ~5 MB; every fixtures/ dir carries a provenance
  README with recorded reference numbers.
- **Design fixtures with detectors built in**: the core t2m fixture
  makes an unweighted mean wrong by 6.7 K by construction, so the
  classic mistake cannot pass silently.
- **Pointwise vs domain-integrated closure**: pointwise holds on any
  spatial subset (tile interiors need no seam operators);
  domain-integrated only on closed domains. Budgets assert pointwise
  on subsets.
- **Measure tolerances before asserting them.** This build's budget
  golden falsified two successive assumed tolerances; the recipe now
  carries the measured absolute criterion with headroom stated. A
  golden's own sanity assertions should be strong enough to catch a
  mis-built fixture (ours did, pre-ship).
- **marimo notes**: constants live inside cells (module-level globals
  are not in the cell graph); no hidden state; app.run() under
  __main__.
